# 智能体界面开发记录

## 1. 目标范围

第一阶段的核心目标，是在 `src/pages/chatbot` 内建立一个可扩展的智能体消息渲染框架，使聊天界面能够从“只渲染文本”升级为“根据后端结构化数据动态分发业务组件”。

本阶段重点完成了以下几件事：

1. 统一消息类型与数据结构。
2. 实现 `MessageRenderer` 核心分发器。
3. 落地 `text / table / chart / form` 四类消息组件。
4. 建立本地 mock provider，支持结构化消息联调。
5. 打通表单提交后的动作链路。
6. 抽离 schema adapter 与 submitAction executor，为下一阶段继续扩展做准备。

## 6. 表单提交动作链路

这是第一阶段里最重要的一条扩展链路。

### 6.1 提交回传

表单提交后，已经不再只是弹出提示，而是能正式触发后续动作。

### 6.2 submitAction 三种模式

当前已支持：

1. `request`
   - 表单提交后继续发起一次新的聊天请求。
2. `insertMessage`
   - 表单提交后直接插入一条结构化消息卡片。
3. `requestAndInsert`
   - 先插入结构化消息，再继续发起聊天请求。

### 6.3 动态模板替换

已支持 `submitAction.message` 内所有字符串字段做模板替换，例如：

```text
{{applicant}}
{{topic}}
{{priority}}
{{planDate}}
```

这意味着表单提交后的结构化卡片，不再是静态内容，而是真正与用户填写结果联动。

### 21.8 智能体特有运行时反馈

当前已经支持：

1. `<think>`
2. 结构化消息
3. 动作按钮

但从“聊天页”走向“智能体界面”，还缺几类关键运行时反馈：

1. 工具调用中的步骤状态
2. 多阶段执行轨迹
3. 可恢复失败节点
4. 中断、重试、继续执行的界面语义
5. 一条消息内多个子任务的局部完成态

这部分的优先级，在真正的 agent 产品中通常高于“继续多加几个消息类型”。

### 21.9 长对话与性能

当前功能已经较全，但真实使用后，性能会很快成为问题。

后续建议重点评估：

1. 长列表虚拟化或分段渲染
2. 图表 / 地图延迟挂载
3. 大型结构化卡片折叠策略
4. 历史消息恢复与滚动锚点保持

原因：

1. 智能体对话通常比普通 IM 更长、更重
2. 一旦消息中混有图表、地图、表格，长对话渲染压力会明显增大
