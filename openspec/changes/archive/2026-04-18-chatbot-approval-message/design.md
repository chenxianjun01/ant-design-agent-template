## Context

当前聊天界面已经支持文本、表格、图表、表单和时间轴消息，但在“审批摘要 / 申请结果 / 待处理单据”场景里，现有类型都不够贴合。表格过重，文本又难以快速扫描关键信息。

## Goals / Non-Goals

**Goals:**

- 新增一个适合审批摘要展示的结构化消息类型
- 保持消息协议、分发器和 mock 扩展方式一致
- 提供本地 mock 样例和自动化测试，便于立即联调

**Non-Goals:**

- 本轮不提供审批按钮和交互回执协议
- 本轮不接入复杂状态流或外部动作执行
- 本轮不通过 SchemaSlot 渲染审批卡片

## Decisions

### 1. 审批卡片先做展示型消息，不做动作协议

原因：当前聊天主线已经具备 submitAction 与表单桥接能力，审批卡片这一轮先解决“摘要展示”问题，不额外引入按钮协议，避免和后续动作语义耦合过早。

### 2. 使用 antd `Card` + `Descriptions` + `Tag` 组合渲染

原因：审批卡片本质上是摘要信息展示，`Card` 可承载标题和状态，`Descriptions` 更适合压缩展示字段清单，`Tag` 适合展示标签或状态补充信息。

### 3. 协议保持轻量字段集合

原因：本轮优先支持标题、描述、状态、摘要、元信息字段和标签，保证协议简单、易 mock、易扩展，后续如果需要审批动作或附件信息，再增量扩展。

## Risks / Trade-offs

- [风险] 只做展示型审批卡片，无法直接覆盖“同卡片内审批”场景
  → 当前目标是先补齐阅读型摘要卡片，动作交互后续单独扩展

- [风险] 字段协议过于自由会降低一致性
  → 先收口到固定的 `fields` 数组结构，避免直接透传任意对象
