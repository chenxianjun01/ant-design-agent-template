## Context

聊天界面已经具备结构化消息主链路，但内容仍以文本和业务表单为主。要达到更完整的智能体工作台体验，还需要支持文件回传、图片预览、音频播放和地图定位。

## Goals / Non-Goals

**Goals:**

- 补齐 `file / image / audio / map` 四类消息
- 保持协议、分发器、mock 和测试的扩展方式一致
- 地图能力使用 `openlayers`

**Non-Goals:**

- 本轮不做文件上传
- 本轮不做音频波形或录音能力
- 本轮不做地图编辑、画图或复杂 GIS 图层管理

## Decisions

### 1. 文本、表格、表单、图表视为已支持能力

原因：这些类型已经在当前代码和测试基线中落地，本轮不重复实现，只把剩余缺口补齐。

### 2. 新增四类消息维持“轻协议 + 专用组件”模式

原因：这与当前 `timeline / approval` 的扩展方式一致，修改面最小，也方便 mock 和契约测试同步扩展。

### 3. 地图使用 OpenLayers 基础底图 + 点位标记

原因：用户明确要求 `openlayers`。本轮先聚焦只读展示能力，支持中心点、缩放和多点 marker，避免过早引入复杂地理编辑逻辑。

## Risks / Trade-offs

- [风险] OpenLayers 在测试环境和 SSR 下更敏感
  → 地图初始化放在 `useEffect` 内，并在组件层做好销毁与降级

- [风险] 富媒体类型增加后协议分支变多
  → 通过统一 `data.ts` 协议校验、`MessageRenderer` 分发和 `service.test.ts` 契约测试来约束漂移
