## Context

聊天页当前状态主要分散在三处：会话列表与活跃会话在页面 state 中，结构化消息在 `useLocalStructuredMessages` 内部 store 中，文本/普通 assistant 响应由 `useXChat` 管理。现在这些状态都只存在内存里，因此页面刷新、重新打开标签页或切换回来后，之前的会话现场会丢失。考虑到执行态运行中的任务已经会被同步进本地结构化消息 store，本轮最合理的方向是增加一个浏览器本地快照层，把页面显示所需的关键状态持久化下来，并在初始化时恢复。

## Goals / Non-Goals

**Goals:**
- 持久化会话列表、活跃会话、聊天消息快照、结构化消息和启动标记
- 页面初始化时恢复这些状态，并保持与现有聊天页逻辑兼容
- 让恢复后的运行中 `agent-execution` 仍能继续接入已有 runtime stream 更新链路
- 保持现有纯对话交互方式，不新增恢复入口卡片或引导页

**Non-Goals:**
- 不做跨设备同步
- 不引入服务端会话存档接口
- 不对本地存储做复杂加密或历史版本迁移系统
- 不保证恢复未完成的网络请求本身，只恢复可展示和可继续更新的本地会话现场

## Decisions

### 1. 使用独立本地快照模块管理序列化，而不是在页面内散写 localStorage

快照读写、默认值回退和 schema 兼容都集中在一个模块中，页面只负责提供当前状态和消费恢复结果。这样后续如果从 localStorage 升级到 IndexedDB，也能保持边界稳定。

备选方案：
- 直接在 `index.tsx` 里多处 `localStorage.setItem/getItem`：短期快，但页面会迅速变成状态和序列化混杂的实现

### 2. 以“聊天消息快照 map + 结构化消息 map”恢复页面显示，而不是尝试操纵 `useXChat` 内部状态

`useXChat` 不是本项目可控状态容器，不适合被外部直接注入恢复值。本轮在页面侧维护一个 `parsedMessageMap` 快照源，作为恢复时的显示基线；当 `useXChat` 输出新的实时消息时，再同步覆盖对应会话快照。

备选方案：
- 强行改写 `useXChat` 内部消息：依赖第三方 hook 内部行为，风险高且不稳定

### 3. 结构化消息 store 支持初始化和对外导出完整 map

当前 `useLocalStructuredMessages` 只能从空 state 启动，不适合恢复。本轮为 hook 增加 `initialMessageMap` 和 `messageMap` 暴露，使页面可以在初始化时灌入恢复值，并在持久化 effect 中写回最新快照。

备选方案：
- 页面单独再维护一份结构化消息镜像：会引入双源状态，容易偏离

## Risks / Trade-offs

- [本地快照与实时消息短时间不一致] → 实时 `useXChat` 输出一旦可用，优先覆盖对应会话快照
- [localStorage 容量有限] → 首轮仅保存聊天页关键工作现场，不做无限历史累积
- [旧格式快照可能解析失败] → 加载时加守卫，失败直接回退默认会话
