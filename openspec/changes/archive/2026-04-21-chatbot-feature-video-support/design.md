## Context

当前 `src/pages/chatbot` 已经形成稳定的结构化消息链路：消息协议定义集中在 `data.ts`，本地 mock 入口在 `service.ts` 与 `mockPayloadFactory.ts`，渲染分发由 `components/MessageRenderer.tsx` 统一调度，不同消息类型再落到专用组件。视频消息应沿用这条链路扩展，避免在页面容器或 provider 中引入额外分支。

现有富媒体类型多为“协议定义 + 专用组件 + mock 样例 + renderer 映射”的增量模式。视频消息也需要采用同样的落点，并明确哪些来源在首轮支持：一类是本地或业务服务可直接访问的 mp4/webm 资源；另一类是能够直接赋给 `<video src>` 的第三方流媒体直链。非直链嵌入页、复杂 DRM 或外部播放器 SDK 不纳入本轮。

## Goals / Non-Goals

**Goals:**

- 为结构化消息体系新增 `video` 类型及对应 TypeScript 协议定义
- 在聊天气泡内直接渲染视频预览，支持封面、标题、时长和格式信息展示
- 保持与现有富媒体一致的分发、mock、测试和联调路径
- 优先复用浏览器原生播放控件，包括原生全屏能力

**Non-Goals:**

- 不接入第三方播放器 SDK，也不做 DRM、鉴权签名或断点续播能力
- 不支持视频上传、转码、字幕轨、清晰度切换和播放埋点分析
- 不为非直链第三方页面做 iframe 嵌入或站外解析兼容

## Decisions

### 1. 协议采用单视频载荷，围绕原生 `<video>` 能力设计

视频消息内容定义为单个视频对象，核心字段为 `url`、`poster`、`title`、`duration`、`format`，必要时可补充描述文本。这样可以直接映射到浏览器原生播放器，避免先引入列表态或复杂媒体编排。

备选方案是沿用图片/音频的数组容器模式，一条消息承载多个视频项。当前业务价值不高，且会增加气泡高度、性能和交互复杂度，因此本轮不采用。

### 2. 交互优先选择气泡内直接预览，而不是默认弹出 Modal

视频消息默认在消息气泡内直接渲染，让用户在会话上下文中完成播放判断，不打断聊天阅读流。全屏能力交给浏览器原生控件处理，必要时用户可以通过播放器进入全屏。

备选方案是点击封面后弹出 Modal。它能减少列表内占高，但会引入额外状态管理、层级问题和移动端遮挡问题，不符合“优先气泡内直接预览”的要求，因此不作为默认方案。

### 3. mock 与真实协议共用同一归一化入口

`video` 类型会像其他结构化消息一样，经由共享的消息协议归一化和 `MessageRenderer` 映射进入渲染链。mock payload 必须使用真实协议字段生成，避免出现只在 mock 中可用、真实返回不可用的分叉。

备选方案是只在 mock provider 中拼出前端专用字段。这样实现快，但会破坏协议一致性，后续接远端返回时容易回归，因此不采用。

### 4. 首轮只支持可直连的视频资源格式

协议中的 `format` 用于声明常见容器格式，如 `mp4`、`webm`，并作为 UI 辅助信息和测试断言依据。播放实现仍基于浏览器对资源的实际支持能力，不做额外解码层。

备选方案是把 `format` 作为必填并据此强校验浏览器兼容性，但这会增加失败分支，且与现有轻协议风格不一致，因此采用“声明式字段 + 原生播放兜底”的模式。

## Risks / Trade-offs

- [风险] 视频资源体积较大，内联播放会增加消息区域高度与加载压力
  → 使用封面图与受控容器高度，默认依赖浏览器懒加载与用户主动播放

- [风险] 第三方流媒体链接未必都是可直连媒体地址
  → 规范中明确只支持可直接赋值给 `<video src>` 的 URL，非直链资源按不支持处理

- [风险] 不同浏览器对编码与全屏 API 的支持差异较大
  → 不封装自定义全屏逻辑，优先依赖原生控件与浏览器能力，测试聚焦协议和渲染存在性

- [风险] 新增消息类型后，mock、类型枚举和 renderer 容易遗漏某个接点
  → 通过 OpenSpec 任务拆分、消息分发测试和 mock 契约测试做回归约束
