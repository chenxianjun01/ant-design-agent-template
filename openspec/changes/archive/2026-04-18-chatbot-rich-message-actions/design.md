## Context

审批卡片已经完成动作协议第一版，说明共享 `submitAction` 执行层不仅适用于表单，也适用于其他结构化消息。富媒体消息当前还停留在展示层，最自然的下一步就是补充动作能力。

## Goals / Non-Goals

**Goals:**

- 为 `file / image / audio / map` 提供统一动作协议
- 复用现有共享 `submitAction` 执行层
- 地图支持 marker 点击回调

**Non-Goals:**

- 本轮不做文件上传
- 本轮不做图片区域标注
- 本轮不做音频编辑或波形分析
- 本轮不做地图绘制和复杂 GIS 交互

## Decisions

### 1. 抽出通用 `IMessageAction`

原因：审批卡片动作和富媒体动作结构一致，抽成通用协议后可以避免同型字段继续复制扩散。

### 2. 文件、图片、音频采用“项内按钮动作”

原因：这些消息通常由单个文件/图片/音频条目触发动作，用每项自带 `actions` 最贴合现有 UI 结构。

### 3. 地图同时支持 marker 点击和列表链接触发

原因：OpenLayers 地图点击适合真实交互，但测试和联调需要更稳定的入口，因此点位列表中的链接按钮会触发同一动作。

## Risks / Trade-offs

- [风险] 动作能力扩散到多消息类型后，状态管理容易重复
  → 通过共享 `useMessageActionExecution` 收口 loading / error / retryable

- [风险] 地图点击事件在测试环境不稳定
  → 地图组件保留列表链接入口，测试优先覆盖稳定断面
